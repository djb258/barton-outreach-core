## Pipeline Position

This repo followed the standard Lovable pipeline:

```
Lovable (initial scaffold) -> GitHub (export) -> Claude Code (ongoing development)
```

The frontend was originally generated by Lovable and exported to GitHub. All ongoing
development, feature additions, and maintenance happen in Claude Code. The repo does
NOT sync back to Lovable -- this is a one-way export, per Lovable doctrine.

The `lovable-tagger` devDependency remains in `package.json` for component attribution
tracking. It tags Lovable-generated components so developers can identify which parts
of the codebase originated from Lovable vs. hand-written code.

## Operational Patterns

### Component Architecture

The frontend uses shadcn/ui conventions:
- Radix UI primitives for accessible, unstyled components
- Tailwind CSS for styling (utility-first)
- `class-variance-authority` for component variants
- `tailwind-merge` + `clsx` for conditional class composition

### UI Features

| Feature | Implementation |
|---------|----------------|
| Theming | `next-themes` (dark/light mode) |
| Resizable Panels | `react-resizable-panels` |
| Drawer | `vaul` |
| Command Palette | `cmdk` |
| OTP Input | `input-otp` |
| Carousel | `embla-carousel-react` |
| Date Picker | `react-day-picker` + `date-fns` |

### Frontend Environment

| Env Var | Purpose |
|---------|---------|
| `NEXT_PUBLIC_SHOW_SUBAGENTS` | Toggle sub-agent visibility in UI |
| `FRONTEND_URL` | Frontend origin (default: `http://localhost:3000`) |
| `ALLOW_ORIGIN` | CORS allowed origin |

### Global Manifest Integration

The `templates/config/global_manifest.yaml` lists `lovable` as a read-access consumer
of the repo dashboard. This means Lovable-generated UI components have read access to
dashboard data but do not write to any backend systems.

## Known Issues

- **No Lovable Cloud backend.** This repo uses Neon PostgreSQL directly, not Lovable
  Cloud or Supabase. The Lovable-generated frontend was rewired to connect to the Neon
  backend after export
- **lovable-tagger in devDependencies.** Kept for component attribution. Safe to remove
  if attribution tracking is no longer needed, but removal requires verifying no build
  scripts depend on it
- **One-way export.** Changes made in this repo cannot be pushed back to Lovable. If
  a new Lovable prototype is needed, it must be a separate project
